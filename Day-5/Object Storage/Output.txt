Claude AI : 
Here's the full breakdown for a LinkedIn-style implementation:

Feature 1 — Hot/Cool/Archive tags on accounts
Both platforms support this, but Azure Blob wins on naming clarity:
Azure Blob has literal tiers called Hot, Cool, and Archive — you set them per-blob or via a lifecycle management policy. For LinkedIn, a lifecycle rule like "if lastModified > 1095 days, move to Archive" maps exactly to the 3-year requirement with zero custom code.
AWS S3 uses S3-Standard, S3-Standard-IA (Infrequent Access), and Glacier — conceptually identical, but the naming is indirect. You'd tag S3 objects with custom metadata to convey "hot/cool/archive" labels to your application, then use lifecycle rules to actually move them between storage classes.

Feature 2 — Frequent reads and append (profile views, connection requests)
This is where the gap is most significant.
Azure Blob has a native Append Blob type. Every AppendBlock API call efficiently adds a segment to the end of the blob — this is exactly how event logs work. Profile view events, connection requests, messaging activity all get appended in real time. The blob stays in Hot tier while active, and a lifecycle rule transitions it to Cool after 90 days.
AWS S3 has no native append operation. Your options are: write a new object per event (expensive at millions of events/day), use multipart upload to simulate appending (complex orchestration), or front S3 with a service like Kinesis Data Firehose that batches writes. All of these add engineering overhead compared to Azure's first-class AppendBlock.

Feature 3 — Archive accounts inactive for 3+ years
Both platforms handle this well via lifecycle policies triggered on lastModified timestamp.
AWS S3 offers three Glacier sub-tiers — Instant Retrieval (milliseconds, higher cost), Flexible Retrieval (3–5 hours), and Deep Archive (12 hours, cheapest). This granularity is an advantage if you want to tune cost vs. recall speed precisely.
Azure Blob Archive rehydrates in up to 15 hours at standard priority, or under 1 hour at high-priority (billed extra). Less granular than Glacier, but the lifecycle policy syntax (daysAfterLastModificationGreaterThan: 1095) is cleaner and maps directly to the "3 year" requirement.

Overall recommendation
Azure Blob Storage is the better fit for this LinkedIn scenario because two of the three features have native, first-class support: the tier labels literally match the requirement, and Append Blob eliminates the most painful engineering gap (frequent event appending).
The only reason to prefer AWS S3 is if your architecture already lives in AWS, or if you specifically need millisecond-level archive restores via Glacier Instant Retrieval — something Azure's Archive tier doesn't match at that speed.
